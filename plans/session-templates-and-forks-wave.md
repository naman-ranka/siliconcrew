# Wave 11 — Session templates & forks (Level 1: bundles, not sessions)

Status: DRAFT (implementation-grade; pending 2nd-agent review, then normal-agent implementation + my adversarial pass)
Supersedes the roadmap plans/session-templates-and-forks.md (that stays as the "why" + future levels).

## The reframe (locked)

**Templates are BUNDLES, not sessions. Only forks are sessions.** A template
is a repo/platform-owned bundle (a workspace snapshot + a small manifest);
you can only FORK it into a user-owned session — so immutability needs zero
enforcement (there is no template-session to mutate). This gets all three
showcases the user wants — the directory, the chat, AND the tool calls — at
Level-1 cost, with checkpoint-continuity cleanly deferred.

## What ships (locked)

- Curated read-only templates as bundles under `examples/<name>/`.
- Fork → new user-owned session (workspace copy + fresh Chat 1 + provenance).
- All three showcases in the fork:
  1. **Directory** — the forked workspace: file tree, runs, reports,
     waveforms all work (run dirs are self-contained, Wave 9).
  2. **Tool calls in Activity** — the bundle's `attempt_events.jsonl` is
     copied, so the fork's Activity dock renders the template's full tool
     trajectory (failures, retries, final pass) with ZERO new machinery.
     Copied `completion.event` markers stop old runs from re-announcing.
  3. **Chat** — each source thread is pre-rendered to a markdown transcript
     FILE in the bundle workspace (`conversations/chat-N-<slug>.md`), so it
     shows in the file tree + artifact viewer, is readable by the user AND
     by the fork's agent ("understand this project"). No checkpoint copy,
     no thread-id rewrite, no read-only chat mode.
- **Deferred (named, not built):** continuing the ORIGINAL conversation
  (LangGraph checkpoint copy + id rewrite) = a future "full fork" wave;
  hosted gallery/versioning/publish-review = a future wave; user
  publish-to-gallery = documented direction only.

## Item 1 — Bundle format + storage

- `examples/<name>/` in the repo:
  - `template.json`: `{ id, name, description, highlights: [str],
    top_module?, platform?, source_note? }` (highlights = the 2-3 metric/
    outcome bullets shown on the gallery card — plain strings, no ambiguous
    verdict parsing).
  - `workspace/`: the snapshot — RTL/spec/tb, `synth_runs/`, `sim_runs/`,
    reports, waveforms, `attempt_events.jsonl`, `conversations/*.md`,
    manifest. Keep bundles LIGHT (small designs; heavy GDS optional and
    documented — big artifacts belong to a later hosted level).
- Ship 1-2 real bundles (e.g. sync_fifo, a tiny ALU) generated by the
  export utility (Item 4) so the format is dogfooded, not hand-authored.

## Item 2 — Fork backend (`src/utils/session_manager.py` + actions)

- `fork_from_template(template_id, user_id) -> session_id`:
  1. Resolve `examples/<template_id>/` (reject unknown → 404).
  2. Allocate a new session id (normal rules; project = none).
  3. Copy `examples/<id>/workspace/` → `workspace/<new_session_id>/`
     (local: filesystem copytree with a size ceiling + file-count guard;
     hosted: DEFERRED — Level-1 targets local/self-host; if the workspace
     provider is cloud, stage the copied dir via the existing `sync` path).
  4. `create_session(...)` metadata (Chat 1 auto-seeded, Wave 9), stamp
     `source_template: {id, name, forked_at}` in session metadata.
  5. Return session id → frontend routes to `/w/<id>`.
- **Half-fork safety:** copy into a temp dir then atomic rename; on any
  failure, clean the temp dir and raise — never leave a partial workspace.
  If metadata insert fails after the dir landed, roll back the dir.
- **Truthfulness:** historical logs/events copied verbatim (they are
  evidence). The ONLY rewrite: none required — run dirs use relative paths
  and thread-agnostic ids; VERIFY no absolute source paths or source
  session id leak into active metadata a tool would consume (manifest,
  run_meta). If any are found, rewrite ONLY those keys, never blind
  search-replace across binaries/logs.
- REST: `GET /api/templates` (list bundles), `GET /api/templates/{id}`
  (manifest + a shallow file preview), `POST /api/templates/{id}/fork`
  (→ `{ sessionId }`). Templates are readable without sign-in (public
  examples); fork requires a signed-in/self-host identity (owns the fork).

## Item 3 — Frontend (Launcher gallery + fork)

- Launcher gains an **Examples** section/tab (or a card row): each bundle
  as a card — name, description, highlight bullets, file/run count. Reuses
  the SessionCard visual language; NO status-dot verdicts (revision 1).
- Card → a lightweight preview (reuse the ThreadDrawer pattern: manifest
  file list + the conversation transcript names + a "what's inside"
  summary from `template.json`). "Fork this example" primary button.
- Fork → `POST .../fork` → `router.push(sessionUrl(sessionId))` — lands in
  the fork; the `source_template` provenance shows a small "forked from
  <name>" chip in the workbench chrome (breadcrumb or session button
  tooltip).
- `api.ts`: `templatesApi.list/get/fork`. Store: `loadTemplates` +
  `forkTemplate`. No new heavy state — templates are read-only list data.

## Item 4 — Export utility (bundle authoring; also the future publish primitive)

- `export_session_bundle(session_id, out_dir)` (a script/CLI, not a hot
  path): copy the session workspace with size/count guards; render each
  chat thread to `conversations/chat-N-<slug>.md` from the thread's
  LangGraph history (reuse the same read path the thread-history endpoint
  uses — `_read_thread_history`); write `template.json` scaffold (curator
  fills description/highlights). Scrub-warn: flag any file that looks like a
  secret/key/env before writing (warn, don't silently include). This is the
  seed for local export/import and, later, hosted publish.
- Transcript renderer: user/assistant turns + tool calls summarized (tool
  name + one-line result), mirroring the Activity card language so the
  markdown reads like the trajectory. Keep it a pure function (testable).

## Item 5 — Tests / gates

- pytest: fork copies workspace + seeds Chat 1 + stamps provenance;
  half-fork rollback (inject a copy failure → no partial workspace, no orphan
  metadata); unknown template → 404; fork ownership (fork owned by the
  caller, not the template); templates list/get shape; transcript renderer
  (threads → markdown, tool calls summarized); export utility round-trips a
  session → bundle → forkable. Copied activity/completion markers don't
  re-announce (the run reads terminal, no new completion event).
- vitest: gallery card render + highlights; fork action routes to /w/{id};
  provenance chip; templatesApi.
- e2e: Launcher Examples → preview → fork → lands in /w/{fork} with the
  file tree + Activity showing the template trajectory + a conversation
  transcript openable in the viewer.
- Gates: pytest · tsc · vitest · Playwright · next build.

## Deferred (documented)
- Level 2: full fork with LangGraph checkpoint copy + thread-id rewrite →
  continue the original conversation live.
- Level 3: hosted gallery, template versioning, object-storage-native large-
  artifact copy, publish/review queue, secret-scrub enforcement (not just
  warn), immutability guarantees, admin moderation.
- User publish-to-gallery flow (controlled, not unrestricted sharing).
